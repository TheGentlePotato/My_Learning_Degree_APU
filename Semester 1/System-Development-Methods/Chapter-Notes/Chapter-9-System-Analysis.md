# Chapter 9 — System Analysis

---

## 1 Introduction to System Analysis

System analysis is the stage in system development where analysts examine the existing system and identify the requirements of the new system. The purpose of system analysis is to understand how the current system works, identify problems and inefficiencies, and determine how the system can be improved.

During system analysis, analysts gather information from users, managers, documents, and existing systems. They study business processes, analyze how data flows through the system, and identify the information needed by users.

The output of system analysis is a clear set of system requirements that will guide the system design phase.

---

## 2 Objectives of System Analysis

The main goal of system analysis is to determine what the new system must accomplish.

| Objective | Explanation |
|---|---|
| Understand the current system | Analyze how the existing system operates |
| Identify system problems | Discover weaknesses and inefficiencies |
| Define system requirements | Determine what the new system must do |
| Improve business processes | Recommend ways to improve system performance |
| Support system design | Provide information needed for the design stage |

System analysis ensures that the development team clearly understands the problem before creating the solution.

---

## 3 Current System Analysis

Current system analysis focuses on understanding how the existing system operates.

Analysts examine several aspects of the system.

| System Component | Description |
|---|---|
| Inputs | Data entered into the system |
| Processes | Activities performed by the system |
| Outputs | Information generated by the system |
| Data storage | Databases or files used to store data |
| Users | People who interact with the system |

Understanding the current system helps analysts determine what changes are necessary.

---

## 4 Identifying System Problems

System analysis identifies problems that exist in the current system.

Common system problems include:

• Slow processing of information  
• Manual procedures that reduce efficiency  
• Duplicate or inconsistent data  
• Limited access to important information  
• Lack of integration between different systems

Once these problems are identified, analysts can propose solutions.

---

## 5 Root Cause Analysis

Root cause analysis is used to determine the underlying causes of system problems.

Instead of only identifying symptoms, analysts try to identify the true cause of the problem.

Example

Problem  
Customer orders are processed slowly.

Possible root cause  
Manual data entry and lack of system automation.

Root cause analysis helps ensure that the new system addresses the real problem.

---

## 6 Business Process Analysis

Business process analysis studies how work is performed within an organization.

Analysts examine workflows and procedures to determine whether they are efficient.

Steps in business process analysis include:

1 Identify business processes  
2 Analyze how tasks are performed  
3 Identify inefficiencies or delays  
4 Recommend improvements

Improving business processes often leads to more effective information systems.

---

## 7 As Is System and To Be System

During system analysis, analysts often describe two versions of the system.

| System Type | Description |
|---|---|
| As Is system | The current system that exists today |
| To Be system | The improved system that will be developed |

The As Is system helps analysts understand current operations.

The To Be system describes how the system will work after improvements are implemented.

This comparison helps identify necessary changes.

---

## 8 Fact Finding Techniques

Fact finding techniques are used to collect information about the system.

| Technique | Description |
|---|---|
| Interviews | Direct discussions with users and stakeholders |
| Questionnaires | Written surveys used to collect information |
| Observation | Analysts observe how users perform tasks |
| Document analysis | Review existing system documents |
| Research | Study external sources or similar systems |

Using multiple techniques improves the accuracy of information collected.

---

## 9 Interviews

Interviews involve direct communication between system analysts and system users.

Advantages include:

• Detailed information  
• Ability to ask follow up questions  
• Direct interaction with stakeholders

Disadvantages include:

• Time consuming  
• Requires careful preparation

Interviews are one of the most effective fact finding techniques.

---

## 10 Questionnaires

Questionnaires are written surveys used to collect information from many users.

Advantages include:

• Fast data collection  
• Low cost  
• Standardized responses

Disadvantages include:

• Limited explanation of answers  
• Possible misunderstanding of questions

Questionnaires are useful when collecting information from large groups of users.

---

## 11 Observation

Observation involves watching users perform their daily tasks.

This method helps analysts understand how work is actually performed.

Advantages include:

• Identifying real work processes  
• Detecting inefficiencies  
• Understanding user behavior

Observation helps analysts identify problems that users may not mention during interviews.

---

## 12 Document Analysis

Document analysis involves reviewing documents used in the current system.

Examples include:

Forms  
Reports  
Manual procedures  
Policies  
User manuals

These documents help analysts understand system operations and data requirements.

---

## 13 Requirements Analysis

After collecting information, analysts analyze the data to determine the requirements of the new system.

Requirements describe what the system must do and how it should perform.

Types of requirements include:

• Functional requirements  
• Non functional requirements  
• User requirements  
• System constraints

Clear requirements help ensure that the system meets user expectations.

---

## 14 Functional Requirements

Functional requirements describe what the system must do.

Examples include:

• The system should allow users to create accounts  
• The system should process online orders  
• The system should generate reports

Functional requirements define the behavior of the system.

---

## 15 Non Functional Requirements

Non functional requirements describe how the system should perform.

Examples include:

• System performance  
• Security requirements  
• Reliability  
• System usability

These requirements ensure that the system operates efficiently and securely.

---

## 16 System Requirements Specification (SRS)

The System Requirements Specification is a formal document that describes the system requirements in detail.

The SRS document usually includes:

System overview  
Functional requirements  
Non functional requirements  
System constraints  
System assumptions

The SRS serves as the foundation for system design and development.

---

## 17 Requirements Prioritization

Not all requirements have the same importance.

Requirements prioritization helps determine which requirements should be implemented first.

Common prioritization categories include:

| Priority Level | Description |
|---|---|
| High priority | Essential system functions |
| Medium priority | Important features but not critical |
| Low priority | Optional features that can be implemented later |

Prioritization helps manage development time and resources.

---

## 18 Modeling the System

System analysts often create models to represent system processes and data structures.

Models help developers understand system behavior and relationships.

Common modeling tools include:

| Modeling Tool | Purpose |
|---|---|
| Data Flow Diagram | Shows movement of data within the system |
| Entity Relationship Diagram | Shows relationships between data entities |
| Use Case Diagram | Shows interactions between users and the system |

These models help visualize how the system works.

---

## Chapter Summary

System analysis is an essential stage in system development where analysts study the current system, identify system problems, and determine the requirements for the new system. Analysts use fact finding techniques such as interviews, questionnaires, observation, and document analysis to gather information about how the system operates. They analyze business processes, identify root causes of system problems, and compare the current As Is system with the proposed To Be system. The collected information is used to define functional and non functional system requirements, which are documented in the System Requirements Specification. Proper system analysis ensures that the development team clearly understands user needs and provides a strong foundation for successful system design and implementation.
