# Chapter 4 — Process Oriented Methodologies

---

## 1 Introduction to Process Oriented Methodologies

Process oriented methodologies focus on **the activities and processes required to build software systems**.

These methodologies emphasize **how work flows through the development process**, rather than focusing only on system structure or people.

In process oriented development, the system is created by identifying tasks, organizing them into workflows, and managing the sequence of activities required to produce the final system.

These methodologies aim to improve productivity by clearly defining development activities.

---

## 2 Key Idea of Process Oriented Development

The core idea is to break software development into **well defined processes**.

Each process represents a step required to transform system requirements into working software.

Typical development processes include:

• Requirement gathering  
• System design  
• Coding  
• Testing  
• Deployment

Process oriented methodologies try to **optimize these processes** to improve efficiency and reduce development time.

---

## 3 Characteristics of Process Oriented Methodologies

| Characteristic | Explanation |
|---|---|
| Process focused | Emphasis on development activities |
| Workflow management | Tasks are organized into structured workflows |
| Process improvement | Continuous improvement of development processes |
| Iterative development | Many process oriented methods support repeated development cycles |
| Productivity improvement | Aim to reduce development time and cost |

---

## 4 Rapid Application Development (RAD)

Rapid Application Development is a process oriented methodology designed to **speed up software development**.

RAD focuses on **rapid prototyping, user feedback, and fast system construction**.

Instead of spending long periods planning and documenting requirements, RAD quickly builds a working prototype of the system.

Users evaluate the prototype and provide feedback, allowing developers to improve the system.

---

## 5 Key Principles of RAD

| Principle | Explanation |
|---|---|
| Rapid prototyping | Quickly build working system models |
| User involvement | Users provide continuous feedback |
| Component reuse | Existing software components are reused |
| Timeboxing | Development tasks must be completed within fixed time limits |
| Parallel development | Multiple system modules developed simultaneously |

---

## 6 RAD Development Phases

RAD development usually follows four major phases.

### Requirements Planning

Developers and users identify system requirements and project scope.

The goal is to determine system objectives quickly.

### User Design

Developers build prototypes of system interfaces and functions.

Users interact with prototypes and suggest improvements.

### Construction

Developers convert prototypes into fully functional software.

Programming and testing occur during this stage.

### Cutover

The final system is tested, installed, and delivered to users.

Training and system deployment also occur during this phase.

---

## 7 Advantages of RAD

| Advantage | Explanation |
|---|---|
| Faster development | Rapid prototyping reduces development time |
| User involvement | Frequent feedback improves system quality |
| Reduced development risk | Problems detected early through prototypes |
| Flexible development | Changes can be implemented quickly |

---

## 8 Disadvantages of RAD

| Disadvantage | Explanation |
|---|---|
| Requires skilled developers | Developers must quickly build working prototypes |
| Requires active user involvement | Without user feedback development may fail |
| Difficult for very large systems | RAD works better for small to medium projects |
| Limited documentation | Rapid development may reduce documentation quality |

---

## 9 Example of RAD Application

Example project

Educational game development.

RAD approach

Iteration 1  
Develop first game level prototype.

Iteration 2  
Improve game mechanics based on teacher feedback.

Iteration 3  
Add additional levels and features.

Each iteration improves the system based on user feedback.

## 10 Rational Unified Process (RUP)

RUP is a structured software development process created by IBM.

It combines structured development practices with iterative development.

RUP divides development into four main phases.

---

## 11 RUP Development Phases

| Phase | Description |
|---|---|
| Inception | Define project goals and scope |
| Elaboration | Analyze requirements and design system architecture |
| Construction | Develop software components |
| Transition | Deliver system to users and perform deployment |

Each phase may include multiple iterations.

RUP focuses on risk management and architecture design.

---

## 12 Spiral Model

The Spiral Model combines elements of **Waterfall and iterative development**.

The model was designed to handle complex and high risk projects.

Development progresses through a spiral shaped process where each cycle represents a development phase.

Each spiral loop contains four activities.

---

## 13 Spiral Model Activities

| Activity | Description |
|---|---|
| Planning | Identify objectives and requirements |
| Risk analysis | Evaluate potential risks |
| Engineering | Develop and test system components |
| Evaluation | Users review the system |

This cycle repeats until the system is completed.

---

## 14 Advantages of Spiral Model

| Advantage | Explanation |
|---|---|
| Strong risk management | Risks identified early |
| Iterative development | System improved gradually |
| Flexible design | Changes can be incorporated easily |
| Early user evaluation | Users provide feedback frequently |

---

## 15 Disadvantages of Spiral Model

| Disadvantage | Explanation |
|---|---|
| Complex management | Requires experienced project managers |
| High cost | Risk analysis and iteration increase cost |
| Not suitable for small projects | Better for large and complex systems |

---

## 16 Extreme Programming (XP) Overview

Extreme Programming is an Agile methodology focused on **high quality coding practices and continuous feedback**.

XP emphasizes technical practices that improve software quality.

Important XP practices include

• Pair programming  
• Test driven development  
• Continuous integration  
• Small releases  
• Simple design

XP aims to improve code quality and reduce development errors.

---

## 17 Prototyping in Process Oriented Development

Prototyping is an important concept in many process oriented methodologies.

A prototype is an **early model of the software system used to test ideas and gather feedback**.

Types of prototypes include:

| Prototype Type | Description |
|---|---|
| Throwaway prototype | Temporary model used to understand requirements |
| Evolutionary prototype | Prototype gradually improved into final system |
| Incremental prototype | Multiple prototypes combined to form final system |

Prototypes help developers identify problems early.

---

## 18 When to Use Process Oriented Methodologies

Process oriented methodologies are suitable when:

| Situation | Reason |
|---|---|
| Development speed is important | RAD provides rapid development |
| User feedback is required | Prototypes allow early evaluation |
| System complexity is high | Structured processes help manage tasks |
| Risk management is necessary | Spiral model identifies risks early |

Rapid Application Development  
Rational Unified Process
